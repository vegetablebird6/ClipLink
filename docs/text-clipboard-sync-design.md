# 文本剪贴板同步设计

## 背景

ClipLink 当前最核心的价值是跨设备快速同步剪贴板文字内容。文件、图片、文档预览和 Blob 存储会显著扩大产品边界，也会引入上传、下载、权限、存储清理、预览安全等复杂度。

因此当前路线收敛为：优先做好文字类剪贴板同步，不做文件同步系统。

## 产品边界

当前支持的剪贴板内容类型：

- `text`：普通文本
- `link`：链接
- `code`：代码片段
- `password`：密码、token、密钥等敏感文本

富文本不新增独立类型。富文本仍然归属于 `text`，通过 `content_html` 和 `content_format` 保存 HTML 版本，`content` 始终保留纯文本 fallback。

## 非目标

当前阶段不做：

- 图片上传、预览和同步
- 文件上传、下载和同步
- PDF、Word、Excel、CSV 等文档解析或预览
- Blob 存储、文件 hash 去重和对象存储接入
- 内容版本管理
- 多人协同编辑

这些能力如果以后要做，应该作为独立专项重新设计，不应继续挤进 `ClipboardItem.content` 字段。

## 数据模型原则

继续使用轻量的 `ClipboardItem` 模型：

```text
id
channel_id
content
content_html
content_format
content_hash
type
title
device_id
device_type
favorite
created_at
updated_at
```

关键约束：

- `type` 只允许 `text/link/code/password`。
- `content` 是主数据，始终存纯文本。
- `content_html` 只用于富文本格式保真。
- `content_format` 只允许 `plain/html`。
- `content_hash` 基于 trim 后的纯文本 `content` 计算，用于重复内容清理。
- 敏感内容仍然作为文本保存，后续增强应围绕提示、隐藏展示和加密展开。

## API 原则

保持现有 `/api/clipboard/*` API，不引入 `/api/items/*` 或内容资产 API。

推荐保留的能力：

- 保存文本剪贴板内容
- 获取当前内容
- 历史记录
- 收藏
- 搜索
- 按类型筛选
- 按设备筛选
- 重复内容清理
- 同步事件历史

不提供上传、下载、预览文件等接口。

## 前端体验原则

首页继续是剪贴板工作台，而不是内容资产管理器。

核心体验：

- 自动读取系统剪贴板中的纯文本或 HTML 富文本。
- 手动添加文本、链接、代码、密码。
- 卡片按文字内容渲染，不展示图片/文件入口。
- 搜索和类型筛选只围绕文字类型。
- 密码类内容默认隐藏，用户可手动显示。
- 富文本在支持的浏览器中优先按 HTML 写回剪贴板，不支持时降级为纯文本。

## 后续可做的文字专项

更适合当前产品边界的后续增强：

- 代码片段语法高亮和语言识别优化
- 标签或分类
- 敏感内容检测和保存前确认
- 密码类内容的本地隐藏、复制保护或端到端加密
- 更稳定的多设备同步事件游标
- 桌面端后台监听剪贴板
- 移动端手动粘贴体验优化

## 决策

在项目正式上线前，直接移除未落地的 `image/file/other` 剪贴板类型入口，避免 API、UI 和文档暗示已经支持文件同步。

如果未来确实要做文件或文档同步，应先重新评估产品定位，再单独设计存储、权限、下载和清理模型。
